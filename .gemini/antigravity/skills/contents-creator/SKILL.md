# contents-creator

원본 1개 → 채널별 완성형 콘텐츠 N개를 만드는 OSMU 에디터 워크플로우.

핵심 원칙: **채널마다 알고리즘과 독자 행동이 다르다.** Instagram은 저장·공유,
Threads는 대화·공감, X는 압축·확산. 같은 콘텐츠를 그대로 복사해 올리면 어느 채널에서도
제대로 안 먹힌다. 각 채널의 행동 목적에 맞게 처음부터 다시 써야 publish-ready라고
부를 수 있다.

## 입력 두 가지 패턴

| TYPE | 입력 형태 | 첫 처리 |
|------|----------|--------|
| A | `.md` 원본 파일 — 블로그 초안, 리서치, 인터뷰, 강의노트, 유튜브 스크립트 | 그대로 STEP 1 진입 |
| B | 대화 형식 한 줄 — 예: "AI 시대 검색 유입 감소와 브랜드 전략" | 핵심 주제·타겟·문제·목적·메시지·키워드·AEO 질문을 먼저 추론 후 STEP 1 진입 |

## 출력 파일 규칙

### 파일명 (고정)

| 파일 | 필수/선택 | 설명 |
|------|---------|------|
| `instagram_post.md` | 필수 | Instagram 피드·캐러셀 본문 |
| `threads_post.md` | 필수 | Threads 포스트 |
| `x_post.md` | 필수 | X.com 단일 포스트 + Thread 버전 |

채널이 추가될 경우에도 같은 규칙(`<channel>_post.md`)을 따른다.

### 저장 위치 — 결정 우선순위

1. **사용자가 폴더를 명시했으면 그 경로를 그대로 사용** (슬러그 변환 없이)
2. 명시가 없으면 작업 폴더 하위에 `contents/<slug>/` 자동 생성

### slug 규칙

- 형식: **`NN-<핵심주제>`** — 2자리 0-padded 넘버링 + 영문 핵심주제
- 예: `01-agent-automation`, `02-context-engineering`, `03-prompt-vs-context`
- **NN 결정**: `contents/` 폴더 안에 이미 있는 NN 중 **최댓값 + 1**. 비어 있으면 `01`부터 시작
- 핵심주제 부분: 영문 소문자 · 숫자 · 대시(`-`)만 사용
- 한국어 주제는 핵심 키워드를 영문화 (예: "AI agent 자동화" → `agent-automation`)
- 전체 길이 60자 이내, 시작·끝 `-` 금지
- 자동 생성한 슬러그는 **채팅에 1줄로 보고**하고, 사용자가 마음에 안 들면 다른 이름으로 바꿀 수 있음을 안내
- 사용자가 폴더명을 직접 명시한 경우 그 이름을 그대로 사용 (NN 자동 부여 안 함)

### 재실행 시 동작 (중요 — 무단 덮어쓰기 금지)

같은 폴더가 이미 존재하면 **자동으로 덮어쓰지 말 것.** 다음 3선택지를 1회 묻고 사용자 응답을 받은 뒤 진행한다:

```
contents/01-agent-automation/ 폴더가 이미 있습니다. 어떻게 할까요?
  [1] 덮어쓰기 — 기존 파일을 새 결과로 교체 (이전 버전은 git history에 남음)
  [2] 새 폴더로 분기 — contents/01-agent-automation/v2/ 식으로 생성
  [3] 취소
```

- **[1] 덮어쓰기 (기본 권장)**: 이 리포는 git 저장소이므로 이전 버전은 `git log -- <파일>`로 복원 가능. 후속 `buffer-publish` 워크플로가 "최신본" 경로를 안정적으로 참조하려면 같은 경로 유지가 유리하다.
- **[2] 새 폴더로 분기**: 비교가 필요할 때만 사용자가 명시적으로 선택. 폴더명은 `v2`, `v3` 식으로 자동 증가시키되, 사용자가 다른 이름을 주면 그걸 우선한다.
- **[3] 취소**: 어떤 파일도 생성하지 말 것.

저장 완료 후 저장된 경로(작업 폴더 기준 상대경로 또는 절대경로)를 채팅에 보고하고,
필요하면 후속 `buffer-publish` 스킬에 그대로 넘길 수 있다는 점을 1줄로 안내한다.

## 3단계 워크플로우

### STEP 1 — 콘텐츠 분석

생성 전에 반드시 먼저 수행한다. 분석을 건너뛰면 채널 차별화도 SEO/AEO 반영도
불가능하다. 단, 사용자의 목적은 결국 콘텐츠 파일이므로 분석은 **짧은 불릿 형태**로
정리해 보여주고 곧장 다음 단계로 진행할 것 — 긴 분석 보고서로 시간을 끌지 말 것.

분석 항목:

1. **핵심 주제** — 한 문장 정의
2. **타겟 독자** — 직군 / 상황 / 관심사
3. **콘텐츠 목적** — 다음 중 하나 이상 선택:
   브랜딩 / 정보전달 / 신뢰 구축 / 저장·공유 유도 / 커뮤니티 확산 / 리드 생성 / 판매 유도
4. **독자의 Pain Point** — 무엇을 모르거나 못 해결하고 있는가
5. **핵심 인사이트 3~5개** — 콘텐츠가 전달해야 하는 깨달음
6. **검색 키워드** — Primary 1개 + Secondary 2~5개
7. **AEO 질문 1~3개** — AI 답변 엔진(Perplexity · ChatGPT 검색 · Gemini 등)이
   인용할 만한 검색형 질문 문장

> AEO 질문 예시
> - "왜 AI 시대에 SEO 트래픽이 감소하고 있을까?"
> - "브랜드는 검색 감소 시대에 어떻게 직접 도달을 만들 수 있을까?"

### STEP 2 — OSMU 유형 판단

주제 성격에 맞는 콘텐츠 유형을 선택한다. **복수 선택 가능, 선택 이유 명시 필수.**

| 유형 | 적합한 상황 |
|------|-----------|
| 교육형 | 새 개념·툴·프로세스를 초보자에게 처음 소개할 때 |
| 체크리스트형 | 단계·항목이 명확하고 저장 가치가 높을 때 |
| 비교형 | A vs B — 도구·전략 선택을 도와줄 때 |
| 반박형 | 통념·미신을 깨야 할 때 (논쟁성 ↑) |
| 실패사례형 | 신뢰 구축 · 공감대 형성 · "나도 그랬다" |
| 스토리형 | 감정 전달 · 인간미 · 긴 체류가 필요할 때 |
| 실험형 | 실제로 해본 데이터·결과 공유 (가장 강한 신뢰 신호) |
| 트렌드형 | 시의성 있는 변화·신호 해석 |
| 프레임워크형 | 머릿속에 남는 모델·매트릭스·공식 형태 |

### STEP 3 — 채널별 콘텐츠 생성

**원칙: 채널마다 처음부터 다시 쓴다.** 같은 단락·문장이 두 채널에 동시에 등장하면
복붙으로 간주, 다시 쓸 것. 각 파일은 아래 구조를 **정확히** 따른다 (헤딩 레벨까지).

---

#### [A] Instagram → `instagram_post.md`

**목적:** 저장(Save) · 공유(Share) · 프로필 방문

| 항목 | 값 |
|------|-----|
| 피드·Reel 캡션 권장 | **150 ~ 400자** |
| 피드·Reel 캡션 최대 | 500자 (이 이상은 모바일 가독성 급락) |
| 시스템 한도 | 2,200자 |
| **첫 125자 룰** | Hook + 핵심 결론을 첫 125자 안에 — 그 뒤는 "...더 보기" 컷으로 가려짐 |
| 캐러셀 텍스트형 long-form (선택) | 600 ~ 1,500자 — 사용자가 "캐러셀로 가자"고 명시할 때만 |

> 글자수는 **공백·줄바꿈 포함**으로 카운트한다 (Instagram 시스템 방식과 동일).
> 한국 실제 운영 패턴은 짧은 캡션이 표준 — 1,000자 이상은 캐러셀 슬라이드 없이
> 텍스트로만 정보를 풀어내는 특수 케이스다. 사용처 확인 없이 long-form으로 가지 말 것.

**구조 규칙:**
1. **첫 125자가 전부** — Hook + 핵심 결론을 그 안에 다 담을 것 ("더 보기" 컷 전 노출)
2. 짧은 문장 + 줄바꿈 적극 사용 (모바일 가독성)
3. CTA — 저장 · 공유 · 프로필 방문 중 하나를 명시
4. 해시태그 — 본문 끝, 5~10개 권장 (30개까지 가능하나 효과 감소)
5. **캐러셀 long-form 변종에서만** — 체크리스트·프레임워크 구조로 정보를 풀어내고, 슬라이드와 1:1 매핑할 것

**SEO/AEO 규칙:**
- 핵심 키워드를 첫 1~2줄에 자연스럽게 삽입
- 검색형 질문을 본문에 1번 포함 (AI 검색 인용 유도)
- 키워드 반복 과잉 금지 — 억지 스터핑은 알고리즘 불이익

**파일 구조 (정확히 이대로):**

```markdown
# Hook

(스크롤을 멈추는 한 줄)

## Caption

(본문 1,000~1,800자, 줄바꿈 적극 사용)

## CTA

(저장 / 공유 / 프로필 방문 유도 한 줄)

## Hashtags

#태그1 #태그2 #태그3 ...
```

---

#### [B] Threads → `threads_post.md`

**목적:** 대화 · 공감 · 댓글 · 체류 시간

| 항목 | 값 |
|------|-----|
| 단일 포스트 한도 | **500자** (Meta Threads 공식) |
| 권장 길이 | **400 ~ 500자** — 한도까지 풍부하게 활용 |
| 캐스케이딩 (thread) | 500자를 넘기면 thread 배열로 다음 포스트에 이어 쓰기 |

> 무리하게 줄이지 말 것. Threads는 1인칭·서사가 잘 먹히는 채널이라 한도까지 채워도
> 가독성에 거의 영향이 없다. 200~300자는 "성의 없어 보이는" 구간.

**구조 규칙:**
1. 인간적인 관점 — "나는 ~라고 본다", "최근 ~를 봤는데"
2. 생각거리 제공 — 정답보다 질문 / 관점
3. 스토리 허용 — 짧은 일화 OK
4. 지나친 광고 톤 금지 (Threads는 마케팅 거부감이 큼)
5. **마지막에 질문 필수** — Discussion Question으로 댓글 유도

**SEO/AEO 규칙:**
- 질문형 문장을 본문에 1회 이상 포함
- conversational tone 유지 — 격식체로 가면 댓글이 안 달림

**파일 구조 (정확히 이대로):**

```markdown
# Hook

(첫 줄에서 관점/주장을 던진다)

## Body

(본문 300~700자, 1인칭 허용)

## Discussion Question

(댓글을 부르는 질문 1개)

## CTA

(팔로우 / DM / 링크 등 — 광고 톤 X)
```

**캐스케이딩(연재) 패턴 — 선택**

다음 두 조건 중 하나면 단일 포스트 대신 캐스케이딩으로 분해:
1. 본문이 500자 한도를 넘어가는 경우 (단일 불가)
2. 사용자가 "연재" · "캐스케이딩" · "thread"로 발행하길 명시한 경우

분해 원칙:
- 첫 포스트(= 발행 시 `text` 필드) = Hook + 핵심 주장
- 후속 포스트 2~5개 = 각 1개의 인사이트 또는 짧은 일화
- 마지막 포스트 = Discussion Question + CTA로 마무리
- **각 포스트가 독립적으로 500자 한도** (합산 아님)

**파일 구조 (캐스케이딩):**

```markdown
# Hook

(스레드 진입을 끄는 첫 줄)

## Body

(첫 포스트 본문 — 발행 시 text 필드로 들어감)

## Thread

1. (두 번째 포스트, 300~500자)
2. (세 번째 포스트, 300~500자)
3. (마지막 포스트 — Discussion Question + CTA 합쳐서)
```

발행 시 `Body` → `text` 필드, `Thread` 항목들 → `metadata.threads.thread` 배열에
순서대로 적재. `Thread` 섹션이 없으면 단일 포스트로 발행.

#### [C] X.com → `x_post.md`

**목적:** 확산 · 리포스트 · 전문가 포지셔닝

X는 **weighted character** 시스템을 쓴다. 영문·숫자 = **1 weight**, 한국어·CJK·이모지 = **2 weight**, URL = 23 weight 고정.

| 항목 | 값 |
|------|-----|
| 무료 계정 단일 포스트 한도 | **280 weight** (한국어 ≈ 140자, 영문 ≈ 280자) |
| 한국어 본문 권장 | **120 ~ 135자** (인용 RT 멘트 공간 여유) |
| Thread 포스트 수 | 5 ~ 10개 |
| Thread 각 포스트 | 같은 280 weight 한도 적용 |
| 유료 (X Premium) | longform 약 25,000자 — 단, 무료 사용자 피드에서는 280 weight에서 잘려 노출 |

> 한국어가 더 짧다고 답답해하지 말 것 — X는 압축이 미덕인 채널이다.
> 단언형·숫자·구체 사실을 압축할수록 인용 가능성이 올라간다.

**구조 규칙:**
1. **주장형 시작** — 두루뭉술한 도입 금지, 첫 문장이 곧 결론
2. 한 포스트 = 하나의 인사이트 (압축적)
3. 숫자·구체 사실 활용 → 인용 가능성 ↑
4. 논쟁성 허용 — 호불호 갈리는 단언이 리포스트를 유발
5. 형용사·부사 제거, 명료한 단언형
6. 마지막 포스트에 공유 유도 한 줄

**SEO/AEO 규칙:**
- 핵심 키워드를 첫 포스트 안에 명확히 포함
- 인용 가능한 1문장 압축 표현을 **Key Takeaway**로 분리 — AI 답변 엔진 발췌 대비
- 단언형 문장이 검색 인용에 가장 잘 걸린다

**파일 구조 (정확히 이대로):**

```markdown
# Single Post
```

## 출력 규칙 — 10가지

1. 반드시 **채널별로 별개 Markdown 파일**로 출력
2. **publish-ready** — 그대로 복사하면 게시 가능해야 함 (자리표시자 금지)
3. 채널별 **문체·구조 차별화** — 같은 표현이 두 채널에 동시에 등장하면 안 됨
4. 같은 표현 반복 금지 (한 파일 안에서도)
5. **SEO/AEO 반영** — 키워드 · 검색 질문 · 인용 가능한 압축 문장
6. 사용자의 **원본 의도 유지** — 새 주장을 마음대로 추가하지 말 것
7. 내용이 부족하면 추론으로 보강 가능 — 단, 추측임을 채팅 응답에 명시
8. 과장·허위 표현 금지 ("100% 보장", "유일한 방법", "검증된" 등)
9. **출력 파일 구조를 깨지 말 것** — 위 markdown 헤딩 구조 그대로
10. 채널별 글자수 SPEC 준수 — 초과 시 줄이고, 부족하면 늘릴 것

## 자주 빠지는 함정

- **복붙 금지를 어기는 가장 흔한 방식:** Instagram Caption 본문을 그대로 Threads
  Body에 옮기고 마지막 질문 한 줄만 다는 것. 두 채널은 문체·길이·호흡이 다르다.
  Threads는 1인칭 conversational, Instagram은 저장형 정보 구조 — 다시 써야 한다.
- **Hook을 도입부로 착각:** Hook = 첫 한 문장 자체가 결론/충격/질문이어야 한다.
  "오늘은 ~에 대해 이야기해보려고 합니다" 같은 도입은 Hook이 아니다.
- **X Thread 1번 포스트를 도입으로 낭비:** 1번이 곧 가장 강한 단언이어야 리포스트가
  나간다. 인사·자기소개로 시작하면 노출이 거기서 끝난다.
- **AEO 질문을 SKILL 안에서만 쓰고 본문엔 안 넣음:** STEP 1에서 도출한 AEO 질문은
  최소 한 채널의 본문 안에 자연 문장으로 녹여야 의미가 있다.
